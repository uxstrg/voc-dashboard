## 1. 분석 파이프라인 개요

고객의 목소리(VoC)가 입력되면 아래의 7단계 파이프라인을 거쳐 정량화된 진단 결과로 변환됩니다.

```text
Raw VoC 텍스트 입력
    ↓
Step 1. 원자적 이슈 분해 (Context Atomization)
    ↓
Step 2. 피드백 유형 분류 (Feedback Type)
    ↓
Step 3. 진단 도메인 확정 (Domain Assignment)
    ↓
Step 4. 품질 속성 매핑 (Attribute Mapping)
    ↓
Step 5. 감성 및 심각도 판단 (Scoring)
    ↓
Step 6. 기대 수준 판단 (Expectation Baseline)
    ↓
Step 7. Gap 분석 및 상태 판정
    ↓
DiagnosedVoC JSON 결과 → 사이트에 하드코딩
```

---

## 2. Step 1. 원자적 이슈 분해

**목적**: 하나의 VoC 안에 섞인 여러 이슈를 독립 단위로 분리  
**분해 기준**: 문장 단위가 아닌 **[대상(Object) + 경험(Experience)]** 맥락 단위. 고객의 반응·결론·행동 의도는 별도 이슈로 분리하지 않고 맥락으로 흡수한다.

### 예시
```text
원문: "11시 유플 다이소 안됐다 생각해보니 11시 정각에 들어가니까 안되는거임,, 에잇 10초남기고 들어가야게따"

❌ 잘못된 분해:
- 이슈 1: [이벤트] + [접속 실패]
- 이슈 2: [사용자] + [우회 시도] ← 고객의 반응을 별도 이슈로 분리

✅ 올바른 분해:
- 이슈 1: [이벤트] + [정각 접속 실패]
  ├── 이슈 맥락 : 접속 실패
  └── 고객 상황 신호 : { situation: "이벤트 정각 접속", loyalty: null }
```

**가드레일**:
* 고객이 명시하지 않은 원인을 추론하여 이슈를 만들지 않는다.
* 고객의 반응·결론·우회 행동은 이슈 맥락 또는 고객 상황 신호로 흡수한다.
* **raw 파일(`voc_raw_*.txt`)에 포함된 모든 항목은 수집 단계에서 이미 노이즈 필터링이 완료된 상태다. 진단 단계에서 "이게 VoC인가", "진단할 만한가"를 재판단하지 않는다. 파일 안의 전 항목을 예외 없이 진단한다.**

## 6. Step 5. 스코어링 공식

### 실제 경험 점수
$$Score_{reality} = 80 + (Sentiment \times Severity)$$

### 감성 점수 (Sentiment)
| 등급 | 값 | 판단 기준 |
| :--- | :--- | :--- |
| 매우 긍정 | +10 | 극찬, 강한 만족 표현, 추천 의사 |
| 긍정 | +5 | 만족, 좋다는 표현, 기대 이상 |
| 중립 | 0 | 사실 서술만, 감정 없음 |
| 부정 | -20 | 불만, 실망, 불편함 표현 |
| 매우 부정 | -40 | 욕설·격앙, 이탈 의사, 브랜드 비난 |

### 심각도 배수 (Severity)
| 등급 | 배수 | 판단 기준 |
| :--- | :--- | :--- |
| 일반 | ×1.0 | 단순 불편 언급 |
| 반복·우회 | ×1.5 | "또", "계속", "몇 번째", 우회 시도 언급 |
| 이탈·기만 | ×2.0 | "해지할 것", "환불", "사기", "거짓말" |

### 심각도 판정 키워드
| 등급 | 키워드 |
| :--- | :--- |
| 반복·우회 | 또, 계속, 매번, 몇 번째, 항상, 반복, 여러 번, 우회, 다른 방법 |
| 이탈·기만 | 해지, 환불, 사기, 거짓말, 타사로, 바꿀까, 탈출, 포기, 이탈, 속았다 |

### 계산 예시
```text
"앱이 또 튕겼어요. 진짜 해지할까봐요"
→ 매우 부정(-40) × 이탈·기만(2.0) = -80
→ Score_reality = 80 + (-80) = 0점

"생각보다 빠르네요"
→ 긍정(+5) × 일반(1.0) = +5
→ Score_reality = 80 + 5 = 85점
```

**상대적 진단 원칙**: 동일한 장애라도 High 등급 고객의 Gap은 더 심각하게 표현한다.

## 8. Step 7. Gap 분석 및 상태 판정

$$Gap = Score_{reality} - Score_{exp}$$

| 상태 | 조건 | 컬러 | 의미 |
| :--- | :--- | :--- | :--- |
| **Exceeding** | Gap > 5 | Green | 기대를 뛰어넘는 경험 제공 중 |
| **Alignment** | -5 ≤ Gap ≤ 5 | Yellow | 기본 품질 안정적 유지 중 |
| **Gap** | Gap < -5 | Red | 고객 실망 및 이탈 위험 구간 |

## 10. 도메인 점수 집계 로직

개별 이슈의 Gap 점수를 도메인 단위로 집계:

```typescript
function calcDomainScore(issues: DiagnosisIssue[], domain: string): number {
  const domainIssues = issues.filter(i => i.domain === domain);
  if (domainIssues.length === 0) return 80;
  const avgGap = domainIssues.reduce((sum, i) => sum + i.gap, 0) / domainIssues.length;
  return 80 + avgGap;
}
```

## 11. 종합 점수 집계 로직

4개 도메인 점수를 가중 평균하여 전체 브랜드 건강 상태를 단일 수치로 표현.

```typescript
function calcOverallScore(issues: DiagnosisIssue[]): number {
  const domains = ['전략', 'UX', '운영', '기술'];
  const domainScores = domains.map(d => ({
    domain: d,
    score: calcDomainScore(issues, d),
    count: issues.filter(i => i.domain === d).length
  }));

  const totalCount = domainScores.reduce((sum, d) => sum + d.count, 0);
  if (totalCount === 0) return 80;

  const weighted = domainScores.reduce((sum, d) => {
    const weight = d.count / totalCount;
    return sum + d.score * weight;
  }, 0);

  return Math.round(weighted);
}
```

### 종합 점수 상태 판정
| 상태 | 점수 구간 | 컬러 | 의미 |
| :--- | :--- | :--- | :--- |
| **양호** | 75점 이상 | Green | 전반적으로 기대치 충족 |
| **보통** | 60~74점 | Blue | 일부 영역 개선 필요 |
| **주의** | 45~59점 | Amber | 복수 영역 기대치 미달 |
| **위험** | 44점 이하 | Red | 전반적 브랜드 신뢰 하락 위험 |

> 컬러 코드는 `ui_spec.md` 1항 참고.

**표시 방식**: 진단 헤더에 `종합 점수: 63점 / 100` 및 상태 뱃지 형태로 표시하며, "종합 점수" 레이블을 사용한다.
